run #01 · plateforme end-to-end

Plateforme data des avantages sportifs

PostgreSQLDebeziumRedpandaSparkDelta Lake · MinIOAirflowGreat ExpectationsPrometheus · GrafanaPower BI

Le problème

Sport Data Solution veut récompenser l'activité physique de ses salariés selon deux règles métier : une prime mobilité de 5 % du salaire brut si le trajet domicile-bureau est réalisable activement (≤ 15 km à pied, ≤ 25 km à vélo), et 5 jours de congé bien-être à partir de 15 activités sportives déclarées sur 12 mois. Calculer ça à la main pour toute l'entreprise est ingérable — il faut une chaîne data complète, du signal brut jusqu'au tableau de bord RH.

L'architecture

Postgres (RH + activités)
    │
    │  Debezium CDC — lecture du WAL (pgoutput)
    ▼
Redpanda (topics : cdc.hr.employees, cdc.ops.activities)
    │
    ├──► Spark Streaming ──► Slack   (notification à chaque activité)
    │
    └──► Spark Batch ──► MinIO / Delta Lake ──► Postgres analytics ──► Power BI
                         bronze → silver → gold
                              ▲
                              └── Google Maps API (géocodage + distances)

Orchestration : Airflow (DAG quotidien  bronze >> silver >> gold)
Qualité       : Great Expectations (conteneur dédié, 5 expectations)
Monitoring    : Prometheus + Grafana

Deux flux complémentaires cohabitent : le temps réel (un INSERT en base est capté par Debezium, publié sur Redpanda et notifié dans Slack en quelques secondes) et le batch quotidien qui raffine la donnée à travers l'architecture médaillon. Principe structurant : l'historique vit dans Delta/MinIO (source de vérité, time travel), tandis que Postgres analytics ne porte que le dernier état pour la BI.

161salariés traités, 100 % géocodés
255 780 €coût annuel calculé (3,2 % masse salariale)
11services Docker Compose
5/5expectations qualité validées

Points clés

  • CDC plutôt que polling — Debezium lit le WAL de Postgres : aucune charge sur la base source, aucun événement perdu.
  • Idempotence de bout en bout — Delta utilise replaceWhere par date de snapshot, l'export Postgres écrase et tronque : rejouer le pipeline produit exactement le même résultat.
  • Règles métier externalisées — taux de prime, seuils et distances vivent dans une table de paramètres : modifier le taux à 8 % et relancer le DAG suffit à tout recalculer, sans toucher au code.
  • Qualité conteneurisée — les 5 expectations (volumétrie, bornes de coûts et de salaires, référentiel des services, intégrité des clés) tournent dans un conteneur dédié, reproductible partout.
  • Monitoring pragmatique — Redpanda n'expose pas nativement le lag consumer ; le dashboard Grafana surveille donc le débit produce/consume et le nombre de consumers actifs, métriques réellement disponibles et tout aussi révélatrices.

Ce que j'en retiens

C'est le projet qui assemble tout : capture, streaming, lakehouse, orchestration, qualité, monitoring, restitution. La leçon principale est architecturale — séparer la source de vérité (Delta, historisée) de la couche de service (Postgres, dernier état) clarifie les responsabilités et évite qu'un besoin BI ne dicte la structure du lakehouse. Et l'idempotence n'est pas un luxe : c'est ce qui permet de relancer un pipeline un dimanche soir sans transpirer.

run #02 · rag en production

Assistant RAG Puls-Events — du POC au MVP

Mistral AIQdrantLangChainRAGASsmolagents · TavilyStreamlitDocker

Le problème

Le POC avait prouvé la faisabilité d'un chatbot événementiel RAG, mais avec des limites bloquantes pour un vrai produit : pas de filtrage natif par date, pas de contexte géographique, pas de mémoire de conversation, et des hallucinations occasionnelles. L'objectif du MVP : corriger ces limites structurellement — pas par des rustines — et le prouver par des métriques.

L'architecture

Utilisateur (Streamlit)
      │
      ▼
Détection localisation (IP + sidebar, repli ville → région)
      │
      ▼
Retriever intelligent ──► détection critères temporels / géo
      │                         │
      │                         ▼
      │                 Filtres natifs Qdrant (Range + MatchValue)
      ▼
Base vectorielle Qdrant ◄── Mistral Embeddings
      │
      ▼
Contexte trouvé ? ──non──► Agent web (smolagents + Tavily)
      │ oui
      ▼
Mistral LLM ──► réponse en streaming
      │
      ▼
Monitoring SQLite + feedback utilisateur (dashboard dédié)

Le changement décisif

Dans le POC, le filtrage par date se faisait en Python, en amont de la recherche vectorielle — fragile et coûteux. Dans le MVP, les filtres temporels et géographiques sont exécutés nativement par Qdrant, avant la recherche vectorielle. Le retriever ne voit que des candidats déjà pertinents, ce qui explique en grande partie le bond des métriques.

0,41 → 0,79faithfulness (RAGAS)
0,14 → 0,80context recall
0,32 → 0,57answer correctness

Évaluation menée sur 30 questions fermées construites depuis des chunks réels de la base — le même protocole avant/après, pour une comparaison honnête.

Points clés

  • Mémoire conversationnelle : « et à Versailles ? » fonctionne sans répéter le contexte.
  • Recherche web de secours (agent smolagents + Tavily) quand la base locale ne trouve rien — le chatbot ne répond jamais « je ne sais pas » sans avoir essayé.
  • Monitoring temps réel : taux d'échec, latence et feedback utilisateur dans un dashboard dédié.
  • Limites assumées et documentées : sessions en mémoire (Redis prévu), monitoring SQLite (PostgreSQL au-delà d'une instance), indexation nationale nécessitant un cluster Qdrant.

Ce que j'en retiens

La différence entre un POC et un MVP n'est pas la quantité de features, c'est la mesure. Mettre en place RAGAS d'abord, puis architecturer pour faire monter les métriques, transforme des choix techniques (« filtres natifs ou post-traitement ? ») en décisions objectivables. Documenter les limites restantes fait aussi partie du travail : un MVP honnête vaut mieux qu'une démo qui promet trop.

run #03 · proof of concept

Chatbot RAG Puls-Events — preuve de faisabilité

Mistral AIFAISSLangChainRAGASStreamlitPytest

Le problème

Puls-Events veut savoir si un chatbot peut répondre en langage naturel à des questions sur les événements culturels d'Île-de-France, en s'appuyant sur les données réelles d'OpenAgenda (via OpenDataSoft) — avec des réponses ancrées dans les données, pas inventées. Mission du POC : démontrer la faisabilité et poser une base d'évaluation.

Le pipeline

  • Ingestion — récupération des événements via l'API OpenDataSoft, nettoyage du HTML avec BeautifulSoup.
  • Vectorisation — découpage en chunks et index FAISS avec les embeddings Mistral.
  • Génération — chaîne LangChain orchestrant le retriever et le LLM Mistral, interface Streamlit.
  • Évaluation — dataset d'évaluation enrichi automatiquement (contexte + réponse) et mesuré avec RAGAS ; tests unitaires Pytest sur la couche données.

Les limites — et pourquoi elles comptent

  • L'API limite les résultats à 10 000 événements.
  • Des hallucinations occasionnelles persistent malgré temperature=0.
  • La recherche sémantique ne filtre pas nativement par date — contournement par filtrage Python en amont.

Ces limites, mesurées plutôt que découvertes par surprise, ont directement dicté l'architecture du MVP : migration de FAISS vers Qdrant précisément pour obtenir un filtrage natif temporel et géographique.

Ce que j'en retiens

Un bon POC ne sert pas à impressionner, il sert à apprendre : sa vraie livraison, ce sont les limites identifiées et le protocole d'évaluation. Avoir mis RAGAS en place dès cette phase a permis, au moment du MVP, de prouver chiffres à l'appui que les choix d'architecture amélioraient réellement le système.

run #04 · pipeline ETL

Pipeline ETL météo multi-sources

PythonAirbyteAWS S3MongoDBPytestDocker

Le problème

GreenAndCoop, fournisseur d'électricité coopératif, veut améliorer ses modèles de prévision de la demande en intégrant de nouvelles sources météo : des stations semi-professionnelles (InfoClimat) et des stations amateurs (Weather Underground). Problème : chaque source a son propre format (JSON, Excel), ses propres unités (impériales côté Weather Underground) et son propre schéma. Il faut un pipeline fiable qui rende tout ça exploitable.

L'architecture

Stations météo (InfoClimat / Weather Underground)
        │
        ▼
    [Airbyte]     collecte multi-sources (Excel, JSON)
        │
        ▼
  [Amazon S3]     zone brute  — raw/
        │
        ▼
  [Script ETL]    normalisation, conversion d'unités,
        │         contrôles qualité (validate_raw / validate_processed)
        ▼
  [Amazon S3]     zone transformée — processed/ (JSONL)
        │
        ▼
  [Script MIG]    migration vers MongoDB
        │
        ▼
   [MongoDB]      collections measurements + stations

Le choix d'une zone brute et d'une zone transformée sur S3 n'est pas cosmétique : il permet de rejouer les transformations sans re-collecter, et de tracer chaque anomalie jusqu'à sa donnée d'origine.

Ce que j'ai construit

  • Configuration d'Airbyte pour la collecte des trois sources hétérogènes vers S3.
  • Script ETL Python : harmonisation des schémas en deux modèles cibles (measurements et stations), conversion des unités impériales vers le système métrique.
  • Contrôles qualité générés à chaque exécution, avant et après transformation, sous forme de rapports.
  • Suite de tests Pytest couvrant l'extraction S3 (mocks AWS), les transformations et le pipeline end-to-end.
  • Orchestration ETL puis migration vers MongoDB via Docker Compose : un docker compose up -d suffit.

Ce que j'en retiens

La vraie difficulté d'un pipeline multi-sources n'est pas technique, elle est sémantique : trois stations qui mesurent « le vent » ne mesurent pas la même chose dans les mêmes unités. Définir un schéma cible unique tôt dans le projet, et valider les données contre ce schéma avant et après transformation, m'a évité de découvrir les incohérences au moment de l'analyse — c'est-à-dire trop tard.

run #05 · orchestration

Orchestration d'un pipeline analytique avec Kestra

KestraDuckDBPythonSQLDocker

Le problème

L'entreprise Bottleneck (négoce de vins) reçoit ses données de ventes de trois sources : un ERP, une boutique web, et un fichier de liaison entre les deux. Chaque mois, quelqu'un doit nettoyer, dédoublonner, fusionner et analyser ces fichiers à la main pour produire un rapport de chiffre d'affaires. L'objectif : automatiser entièrement ce processus, avec une exécution planifiée et une gestion d'erreurs digne de la production.

Le pipeline

  • Ingestion — conversion des fichiers XLSX en CSV via Python.
  • Nettoyage & fusion — requêtes DuckDB directement sur fichiers : dédoublonnage de chaque source, puis fusion des trois tables par jointure.
  • Analyse — calcul du CA par produit et du CA total en SQL ; classification des vins par z-score en Python pour repérer les prix atypiques.
  • Restitution — export automatique d'un rapport Excel multi-feuilles (openpyxl).
  • Orchestration — flux Kestra combinant tâches séquentielles et parallèles, trigger temporel, retry automatique et remontée d'erreurs à chaque étape, avec des tests de validation intermédiaires.

Pourquoi DuckDB ?

Pour des volumes mensuels de cette taille, déployer un entrepôt de données serait disproportionné. DuckDB permet d'écrire du SQL analytique performant directement sur les fichiers, sans serveur à maintenir — le bon outil pour la bonne échelle.

Ce que j'en retiens

Les tests de validation intermédiaires entre les tâches du flux sont ce qui distingue un script automatisé d'un pipeline fiable : une erreur détectée à l'étape de dédoublonnage coûte quelques secondes ; la même erreur découverte dans le rapport final envoyé aux équipes coûte la confiance dans tout le pipeline.

run #06 · streaming temps réel

Pipeline streaming de tickets clients

Apache SparkKafka / RedpandaParquetPythonDocker

Le problème

Démontrer, par un proof of concept, qu'un flux de tickets clients peut être traité en temps réel : chaque ticket doit être enrichi (assignation automatique à une équipe support selon son type) et stocké dans un format exploitable pour l'analyse batch, le tout avec une reprise sur panne sans perte ni doublon.

L'architecture

[Producer Python] ──JSON──▶ [Redpanda · topic client-tickets]
                                      │
                                      ▼
                      [Spark Structured Streaming]
                        ├── parsing JSON + schéma explicite
                        ├── logique métier : assignation équipe
                        ├──▶ [Parquet]      /output/parquet/
                        └──▶ [Checkpoints]  /output/checkpoints/

Points clés

  • Schéma explicite appliqué au parsing des messages — pas d'inférence silencieuse qui casserait en production.
  • Tolérance aux pannes via les checkpoints Spark : les offsets Kafka consommés sont persistés, garantissant la reprise automatique sans retraitement.
  • Job batch séparé (spark_insight.py) qui agrège les Parquet produits — illustration du pattern streaming pour l'ingestion, batch pour l'analyse.
  • Infrastructure complète (Redpanda, producteur, consommateur Spark) en un seul docker compose up -d.

Une démonstration vidéo du pipeline en fonctionnement est disponible : voir la démo.

Ce que j'en retiens

En streaming, la question n'est pas « est-ce que ça marche ? » mais « qu'est-ce qui se passe quand ça s'arrête ? ». Comprendre concrètement le rôle des checkpoints — et le vérifier en tuant le consommateur en pleine exécution — m'a appris plus sur les garanties de livraison que n'importe quelle documentation.

run #07 · migration & sécurité

Migration de données de santé vers MongoDB

PythonMongoDBDocker ComposePytestPandas

Le problème

Migrer des données d'hospitalisation de patients depuis des fichiers CSV vers MongoDB — un contexte où l'on ne peut se permettre ni perte de données, ni accès trop permissifs. Le pipeline devait aussi être réutilisable : changer de dataset ne doit demander que la modification du schéma de validation et de la configuration.

L'architecture

docker-compose.yml
 ├── [mongo]       base MongoDB, utilisateurs initialisés
 ├── [migration]   loader → validation → insertion par batch
 ├── [backup]      dump toutes les 2 h, rétention 3 jours
 └── [mongo_test]  base dédiée aux tests Pytest (profil run-tests)

Points clés

  • Moindre privilège — le compte root ne sert qu'à l'initialisation ; l'application utilise un compte readWrite limité à sa seule base.
  • Validation par schéma explicite — contrôle d'intégrité avant insertion, doublons ignorés, vérification post-import.
  • Code modulaire — séparation loader / validation / pipeline / db, ce qui rend le pipeline adaptable à d'autres formats.
  • Tests isolés — Pytest s'exécute contre un conteneur MongoDB dédié, jamais contre la base cible.
  • Sauvegardes planifiées — conteneur de backup avec dump automatique et politique de rétention.

Ce que j'en retiens

Sur des données sensibles, la sécurité n'est pas une couche qu'on ajoute à la fin : la séparation des rôles utilisateurs et l'isolation des environnements de test doivent être présentes dès le premier docker compose up. C'est aussi le projet qui m'a appris à concevoir pour la réutilisabilité — le pipeline a été pensé comme un outil, pas comme un script jetable.

run #08 · nosql distribué

Cluster MongoDB distribué sur données Airbnb

MongoDBPolarsPower BIPython

Le problème

Concevoir une base NoSQL capable de servir 95 885 annonces Airbnb (Paris + Lyon, 74 champs par document) avec haute disponibilité et répartition géographique de la donnée — puis prouver que cette base peut alimenter à la fois des analyses Python et un outil de BI d'entreprise.

L'architecture

        ┌────────────────┐      ┌────────────────┐
        │  rs0 — Lyon    │      │  rs1 — Paris   │
        │  Primary       │      │  Primary       │
        │  Secondary     │      │  Secondary     │
        │  Arbitre       │      │                │
        └───────┬────────┘      └───────┬────────┘
                └─────────┬─────────────┘
                          ▼
                  [Config Server]
                          │
                          ▼
                  [mongos router]
            shard key : city_name + _id
                          │
                          ▼
         [Power BI]  via ODBC + BI Connector

Points clés

  • Pourquoi le document ? Un logement = un document. Données hétérogènes et évolutives, lecture sans jointure, ajout de champs sans migration.
  • ReplicaSet — deux ReplicaSets (dont un avec arbitre), avec test concret de la réplication et du failover.
  • Sharding géographique — clé composée city_name + _id : les chunks de Lyon vivent sur rs0, ceux de Paris sur rs1. La donnée est physiquement proche de sa ville.
  • Analyse — requêtes mongosh (disponibilité, superhôtes, hôtes multi-annonces…) et exploration avec Polars.
  • Restitution BI — chaîne complète ODBC → BI Connector (mongosqld) → Power BI, le proxy SQL qui permet à un outil BI classique de lire du NoSQL.

Ce que j'en retiens

Le choix de la shard key est une décision d'architecture, pas un détail de configuration : sharder par ville rend les requêtes géographiques rapides et la localisation des données explicite, mais engage la structure du cluster pour de bon. C'est le genre de décision qu'il faut savoir justifier — et j'ai appris à le faire sur ce projet.

run #09 · machine learning

Prédiction de consommation énergétique des bâtiments

Scikit-learnBentoMLPandasPoetry

Le problème

La ville de Seattle publie chaque année les données de consommation énergétique de ses bâtiments commerciaux. L'objectif : prédire la consommation réelle d'un bâtiment (SiteEnergyUseWN, en kBtu) à partir de ses seules caractéristiques structurelles, pour anticiper les besoins et identifier les bâtiments énergivores — puis rendre ce modèle utilisable par n'importe quelle application via une API.

La démarche

  • EDA — exploration des distributions de consommation, identification des valeurs aberrantes et des corrélations clés.
  • Feature engineering — 9 variables retenues : type de propriété, quartier, nombre de bâtiments et d'étages, surface, sources d'énergie utilisées (vapeur, électricité, gaz) et âge du bâtiment.
  • Modélisation — comparaison de plusieurs algorithmes en validation croisée ; Random Forest Regressor retenu.
  • Serving — export du modèle dans le store BentoML et exposition via une API REST avec interface Swagger pour tester les prédictions.
0,82R² (validation croisée)
9features retenues
RESTAPI BentoML + Swagger

Ce que j'en retiens

La valeur d'un modèle ne se mesure pas qu'à son R² : un modèle à 0,82 accessible via une API documentée est plus utile qu'un modèle à 0,85 enfermé dans un notebook. Ce projet m'a fait passer du réflexe « notebook » au réflexe « produit » — c'est exactement la frontière entre data science et data engineering.
